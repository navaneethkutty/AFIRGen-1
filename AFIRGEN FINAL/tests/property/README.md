# Property-Based Tests

This directory contains property-based tests using [Hypothesis](https://hypothesis.readthedocs.io/) to verify correctness properties from the design document.

## Overview

Property-based testing generates random inputs to verify that system properties hold across all valid executions. Unlike example-based tests that check specific cases, property tests verify universal invariants.

## Test Coverage

The property tests validate the following correctness properties:

### Property 1: File Format Validation
- **Validates**: Requirements 1.1, 2.1
- **Description**: Verifies that the system accepts only valid file formats (WAV/MP3/MPEG for audio, JPEG/PNG for images) and rejects all others

### Property 2: S3 Encryption
- **Validates**: Requirements 1.2, 2.2, 13.1
- **Description**: Ensures all files uploaded to S3 have SSE-KMS encryption enabled

### Property 10: Embedding Dimensionality
- **Validates**: Requirements 4.4
- **Description**: Verifies that all embeddings generated by Titan have exactly 1536 dimensions

### Property 12: Top-K Search Results
- **Validates**: Requirements 4.8
- **Description**: Ensures vector similarity search returns at most k results, ordered by descending similarity score

### Property 4: Retry Logic with Exponential Backoff
- **Validates**: Requirements 1.6, 2.6, 3.7, 3.8, 4.9, 9.1-9.5
- **Description**: Verifies retry logic with exponential backoff for AWS service failures

### Property 17: Cache Hit Reduction
- **Validates**: Requirements 7.7
- **Description**: Ensures cache hits don't result in additional embedding API calls

### Property 19: API Schema Compatibility
- **Validates**: Requirements 8.2, 8.3
- **Description**: Verifies API request/response schemas are compatible between GGUF and Bedrock implementations

## Running the Tests

### Install Dependencies

```bash
pip install -r tests/requirements-test.txt
```

### Run All Property Tests

```bash
pytest tests/property/ -v
```

### Run Specific Property Test

```bash
pytest tests/property/test_properties.py::test_property_embedding_dimensionality -v
```

### Run with More Examples

By default, Hypothesis generates 50-100 examples per test. To increase coverage:

```bash
pytest tests/property/ -v --hypothesis-seed=random --hypothesis-show-statistics
```

### Run with Specific Seed (for reproducibility)

```bash
pytest tests/property/ -v --hypothesis-seed=12345
```

## Test Configuration

Property tests use Hypothesis settings:
- `max_examples`: Number of random inputs to generate (30-50 per test)
- `suppress_health_check`: Disables certain health checks for async/fixture tests
- `deadline`: Timeout per test case (default: 200ms)

## Understanding Failures

When a property test fails, Hypothesis will:
1. Show the failing input that violated the property
2. Attempt to shrink the input to find the minimal failing case
3. Save the failing example for regression testing

Example failure output:
```
Falsifying example: test_property_embedding_dimensionality(
    text='a'
)
```

## Stateful Testing

The `CacheHitStateMachine` uses Hypothesis's stateful testing to verify cache behavior across sequences of operations:
- Generates random sequences of cache operations
- Verifies invariants hold after each operation
- Tests complex interaction patterns

## Best Practices

1. **Use `assume()` to filter invalid inputs** rather than catching exceptions
2. **Keep properties simple** - test one invariant per property
3. **Use appropriate strategies** - match input types to domain constraints
4. **Document what each property validates** - link to requirements
5. **Run tests regularly** - property tests can find edge cases missed by examples

## Troubleshooting

### Tests are slow
- Reduce `max_examples` in `@settings` decorator
- Use `@settings(deadline=None)` to disable timeouts for slow operations

### Too many invalid inputs
- Refine your Hypothesis strategies to generate more valid inputs
- Use `assume()` to filter, but not excessively (Hypothesis will give up after too many filtered examples)

### Flaky tests
- Ensure tests are deterministic given the same input
- Avoid time-dependent assertions
- Use `--hypothesis-seed` to reproduce failures
